# 项目流程说明

## 1. 项目目标

这个项目用于自动抓取 The Information 的订阅 RSS/Atom feed，把文章整理成结构化 CSV，并按日、按周归档到仓库中。

项目的核心职责分成两层：

1. `fetch_in0407.py`
   负责实际访问订阅 feed、提取 XML、解析条目、清洗正文、输出 CSV。
2. `automate_rss.py`
   负责按时间规则组织日任务和周任务，并在允许的范围内执行 `git add / commit / push`。


## 2. 核心文件与职责

| 文件 | 作用 |
| --- | --- |
| `fetch_in0407.py` | 抓取并解析 The Information feed，生成结构化 CSV |
| `automate_rss.py` | 自动化主流程，负责日抓取、周合并、日志追加、Git 提交 |
| `run_rss_job.sh` | 定时任务入口，加载环境变量并调用虚拟环境 Python |
| `rss_crawl.crontab` | cron 配置模板 |
| `AUTO_COMMIT_RULES.md` | 自动提交白名单/黑名单说明 |
| `AUTO_UPLOAD_LOG.md` | 记录每次自动产出的日文件和周文件 |
| `daily_csv/` | 每日输出目录 |
| `weekly_csv/` | 每周合并输出目录 |
| `state.json` | Playwright 登录态文件，抓取 feed 的必要前提 |
| `debug_response.html` | 调试输出，用于排查抓取失败时拿到的原始内容 |


## 3. 总体执行链路

整体流程如下：

```text
cron
  -> run_rss_job.sh
    -> 加载 THEINFO_USERNAME / THEINFO_PASSWORD
    -> .venv/bin/python automate_rss.py --mode daily|weekly
      -> 日任务: 调用 fetch_in0407.run_fetch() 生成 daily_csv
      -> 周任务: 合并最近 7 个 daily_csv 生成 weekly_csv
      -> 追加 AUTO_UPLOAD_LOG.md
      -> 按白名单执行 git add / commit / push origin main
```

如果是手动执行，也可以跳过 `cron` 和 `run_rss_job.sh`，直接运行 Python 脚本。


## 4. 抓取流程：`fetch_in0407.py`

### 4.1 输入条件

脚本依赖以下输入：

- `state.json`
  Playwright `storage_state`，用于带登录态访问页面。
- `THEINFO_USERNAME`
  HTTP Basic Auth 用户名。
- `THEINFO_PASSWORD`
  HTTP Basic Auth 密码。

这些参数也可以通过命令行显式覆盖，例如 `--state-file`、`--username`、`--password`。


### 4.2 访问 feed 的过程

`run_fetch()` 的内部流程如下：

1. 调用 `fetch_feed_xml()`。
2. 启动 Playwright Chromium 无头浏览器。
3. 基于 `state.json` 创建浏览器上下文。
4. 如果提供了用户名和密码，则追加 `http_credentials`。
5. 打开 `https://www.theinformation.com/subscriber_feed`。
6. 优先从 HTTP 响应文本中提取 `<feed>...</feed>` XML。
7. 如果响应正文里没有直接拿到 XML，再尝试：
   - 从页面 `<pre>` 标签读取；
   - 或从 `page.content()` 里继续提取 XML。
8. 如果仍然提取失败，则把当前页面 HTML 写入 `debug_response.html`，并抛出异常。


### 4.3 XML 解析与正文清洗

拿到 Atom XML 之后，`parse_atom()` 会对每个 `entry` 提取以下字段：

- `标题`
- `时间`
- `作者`
- `链接`
- `内容`

其中正文清洗流程是：

1. 读取 `atom:content` 中的 HTML。
2. 用 `BeautifulSoup` 转成纯文本。
3. 统一压缩多余空行。
4. 遇到以下栏目名时截断正文，避免把推荐内容、活动等尾部信息带入 CSV：
   - `Upcoming Events`
   - `Recommended Newsletter`
   - `New From Our Reporters`
   - `Today on The Information’s TITV`
   - `Today on The Information's TITV`
   - `What We’re Reading`
   - `What We're Reading`


### 4.4 日期过滤逻辑

脚本内部把 feed 中的发布时间统一转换到 `America/Los_Angeles` 时区，再得到 `feed_local_date`。

这意味着：

- `--target-date 2026-05-13`
- 实际筛选的是“美西日期为 `2026-05-13` 的条目”

不是按服务器时区过滤，也不是按北京时间过滤。


### 4.5 CSV 输出

`save_csv()` 会把结果写成 UTF-8 with BOM 的 CSV，表头固定为：

- `标题`
- `时间`
- `作者`
- `链接`
- `内容`

输出路径由 `--csv-output` 决定；目录不存在时会自动创建。


## 5. 自动化流程：`automate_rss.py`

### 5.1 支持的模式

脚本支持三个模式：

- `--mode daily`
  只执行日抓取。
- `--mode weekly`
  只执行周合并。
- `--mode all`
  一定会执行日抓取；如果满足周任务触发条件，还会继续执行周合并。

还支持两个控制开关：

- `--skip-git`
  只生成文件，不执行 `git add / commit / push`。
- `--skip-push`
  会执行 `git add / commit`，但不执行 `git push`。


### 5.2 当前时间与时区

自动化脚本的业务时间统一使用 `Asia/Shanghai`。

`get_now_beijing()` 的规则是：

- 默认取当前北京时间；
- 如果传了 `--now`，则用该时间覆盖，便于测试；
- `--now` 如果没有时区信息，会被直接视为北京时间。


### 5.3 日任务流程

日任务由 `generate_daily_csv()` 实现：

1. 取“当前北京时间减 1 天”作为目标日期。
2. 生成输出路径：`daily_csv/YYYY-MM-DD.csv`。
3. 调用 `run_fetch()`，并把这个目标日期作为 `target_date` 传入。
4. 只保留“美西日期等于该目标日期”的文章。
5. 生成当日 CSV。
6. 往 `AUTO_UPLOAD_LOG.md` 追加一段 `Daily` 记录。
7. 如果没有启用 `--skip-git`，则进入自动提交流程。

例子：

- 北京时间 `2026-05-14 12:00`
- 目标日期是 `2026-05-13`
- 输出文件是 `daily_csv/2026-05-13.csv`


### 5.4 周任务流程

周任务由 `generate_weekly_csv()` 实现：

1. 取“当前北京时间减 1 天”作为结束日期。
2. 再往前推 6 天，得到开始日期。
3. 依次拼出 7 个日文件路径：
   - `daily_csv/start_date.csv`
   - ...
   - `daily_csv/end_date.csv`
4. 合并存在的日文件内容，写入：
   - `weekly_csv/start_date_to_end_date.csv`
5. 如果某些日文件不存在，只打印缺失提示，不会中断周任务。
6. 往 `AUTO_UPLOAD_LOG.md` 追加一段 `Weekly` 记录。
7. 如果没有启用 `--skip-git`，则进入自动提交流程。

例子：

- 北京时间 `2026-05-08 13:00`
- 结束日期是 `2026-05-07`
- 开始日期是 `2026-05-01`
- 输出文件是 `weekly_csv/2026-05-01_to_2026-05-07.csv`


### 5.5 `--mode all` 下的周任务触发条件

`--mode all` 不会无条件执行周任务。

只有在以下条件同时满足时才会跑周合并：

- 当天是周五
- 北京时间小时数 `>= 13`

对应代码函数是 `should_run_weekly()`。

也就是说：

- 平时执行 `--mode all`，只会生成日文件
- 到了周五 13:00 之后执行 `--mode all`，才会额外生成周文件


## 6. 自动提交与推送流程

### 6.1 提交筛选规则

`automate_rss.py` 不会无差别提交所有改动，而是先读取当前仓库状态，再通过白名单筛选可提交文件。

允许自动提交的文件：

- 所有 `.md` 文件
- `daily_csv/` 下的 `.csv`
- `weekly_csv/` 下的 `.csv`

明确禁止自动提交的文件：

- 所有 `.py` 文件
- `state.json`

因此，自动化脚本即使运行时发现代码文件被改动，也不会把这些改动一并提交。


### 6.2 自动提交的具体动作

如果筛选后有允许提交的文件，脚本会执行：

1. `git add -- <allowed paths>`
2. `git commit -m "<message>"`
3. `git push origin main`

提交信息格式如下：

- 日任务：`Add daily RSS CSV for YYYY-MM-DD`
- 周任务：`Add weekly RSS CSV for YYYY-MM-DD to YYYY-MM-DD`

如果本次没有命中白名单变更，脚本会打印 `git no allowed changes`，然后直接结束，不会报错。


## 7. 定时任务入口与 cron 行为

### 7.1 Shell 入口：`run_rss_job.sh`

这个脚本负责两件事：

1. 通过交互式 `bash -ic 'env | grep "^THEINFO_"'` 把当前用户环境中的 `THEINFO_` 变量读出来并导出。
2. 调用虚拟环境中的 Python：

```bash
./.venv/bin/python automate_rss.py --mode "${MODE}"
```

如果用户名或密码缺失，脚本会直接报错退出：

- `Missing THEINFO_USERNAME or THEINFO_PASSWORD`


### 7.2 cron 配置

`rss_crawl.crontab` 当前配置为：

```cron
0 4 * * * /home/ubuntu/work/rss-crawl/run_rss_job.sh daily >> /home/ubuntu/work/rss-crawl/logs/daily.log 2>&1
0 5 * * 5 /home/ubuntu/work/rss-crawl/run_rss_job.sh weekly >> /home/ubuntu/work/rss-crawl/logs/weekly.log 2>&1
```

在服务器使用 UTC 时区时，它的业务含义是：

- 每天 `04:00 UTC` 执行一次日任务，对应北京时间 `12:00`
- 每周五 `05:00 UTC` 执行一次周任务，对应北京时间 `13:00`


## 8. 输出产物

项目运行后会产出以下几类文件：

1. 日 CSV
   路径格式：`daily_csv/YYYY-MM-DD.csv`
2. 周 CSV
   路径格式：`weekly_csv/YYYY-MM-DD_to_YYYY-MM-DD.csv`
3. 自动上传日志
   路径：`AUTO_UPLOAD_LOG.md`
4. 调试页面内容
   路径：`debug_response.html`

```

```


## 10. 关键风险点与排查顺序

项目最容易出问题的环节有四个：

1. `state.json` 失效
   会导致 Playwright 打开的不是可读 feed 页面。
2. `THEINFO_USERNAME` / `THEINFO_PASSWORD` 缺失或错误
   会导致 Basic Auth 失败。
3. 抓到的页面不是 feed XML
   此时需要检查 `debug_response.html`，确认拿到的是登录页、错误页还是别的 HTML。
4. Git 推送失败
   自动流程默认执行 `git push origin main`，远端权限、网络、分支保护都会影响结果。

建议排查顺序：

1. 先看 `logs/daily.log` 或 `logs/weekly.log`
2. 再看 `debug_response.html`
3. 手动运行 `pagetest.py` 验证当前登录态与认证是否仍可访问页面
4. 最后再检查本地 Git 与远端权限


## 11. 一句话总结

这个项目本质上是一个“按北京时间调度、按美西日期筛选内容、按白名单自动提交结果文件”的 RSS 归档流水线。
